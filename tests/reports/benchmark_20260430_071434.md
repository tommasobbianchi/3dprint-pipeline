## Analysis

### Reality check — measured vs analytical estimate

The original analytical estimate was **5–8 s per turn**. Measured P1 median is
**60.78 s** (~10x worse). The breakdown of the gap, based on the running-
process inspection during the benchmark:

- **`uvx mcp-proxy` cold start per invocation**: ~2–3 s. The benchmark spawns a
  fresh `claude --print` for each run with `--no-session-persistence`, so
  every run pays this cost. In real interactive use (one persistent
  `claude` session), this is paid once.
- **Sonnet thinking time per tool call**: ~5–10 s each. P1 used 4 tool calls
  → 20–40 s of LLM time. Bigger context (P3 enclosure) means more thinking
  per call.
- **Onshape REST tool latency**: ~500 ms–2 s per call (sketch/extrude/fillet),
  bigger for FeatureScript. ~5 s total for P1.
- **Microversion polling**: ~0.2–0.5 s tail after Claude returns.

So 60 s ≈ 2.5 s (uvx) + 30 s (LLM × 4 tools) + 5 s (Onshape REST) + 0.3 s (poll)
+ overhead. Plausible, just much worse than the optimistic 5–8 s estimate.

### Why P2–P5 timed out

`claude_cli_failed` with elapsed times 240.5–270.5 s = the benchmark's own
240 s timeout. The 270 s cases are runs where the timeout fired but the
process took ~30 s extra to be reaped.

Likely root causes (not investigated in detail):
- **Document state accumulation**: each run leaves features in the Part
  Studio. By P2 run 1, the Part Studio has 8+ unrelated features from
  P1×2. The model spends time inspecting state before acting.
- **Tool retry loops**: when an Onshape feature creation returns an error
  (e.g. invalid sketch reference), Claude often retries with a different
  approach, multiplying tool calls. With ~10 s LLM thinking per call,
  3 retries adds 30 s.
- **No session persistence**: P4 ("modify the previous enclosure +10 mm")
  cannot leverage P3's context — it has to re-discover the feature tree
  from scratch, which is the worst case for an iterative prompt.

### Comparison with the legacy CadQuery iframe path

| Aspect | Legacy iframe | Jarvis MCP (this bench) |
|--------|---------------|--------------------------|
| Median P1 (simple cube) | 25–40 s | **60.78 s** |
| Reliability on simple parts | Worked when service was up | Worked (P1 1/2 — one timeout) |
| Reliability on complex parts | Unknown — fails at Derived feature import (ERROR) | Worked? Unknown — all timed out at 240 s budget |
| Iterative edits | Re-runs full pipeline each turn | Could be near-instant with session persistence (not measured here) |
| Output format | STEP imported as Derived feature (often ERROR) | Native Onshape feature tree |
| Server cost per run | 1 fresh `claude --print` subprocess (~3–5 s startup) + CadQuery exec | 1 fresh `claude --print` subprocess (~3–5 s) + uvx cold (~2–3 s) |
| User Claude subscription | Server's account | User's Max/Pro |

**The MCP path is not faster for one-shot batch invocations.** On a
per-prompt basis it is comparable to the legacy on simple parts and likely
slower on complex ones (assuming the legacy didn't fail on import). The
MCP path wins on:

1. **Reliability** — produces native Onshape features that always
   regenerate, vs Derived-feature `status=ERROR`.
2. **Iterative edits** — when used as designed (one persistent `claude`
   session, not subprocess-per-prompt), the iterative path avoids re-
   running the whole pipeline. Not visible in this benchmark.
3. **Auth model** — user's subscription, not server's.
4. **Composability** — Variable Studios, mates, FeatureScript, render-and-
   verify all available as one toolbox, vs the legacy's CadQuery-only path.

### Recommendations for a better benchmark

1. **Bigger timeout** — 600 s. Complex prompts can legitimately need 200 s+
   on Sonnet.
2. **Reset document between prompt families** — call
   `DELETE /api/v6/documents/d/{did}/elements/{eid}` and re-create a fresh
   Part Studio, so each prompt starts on a clean slate.
3. **Persistent session for iterative prompts** — for P4 ("modify the
   previous"), reuse the P3 session via `claude --resume <session>` instead
   of a fresh subprocess.
4. **Capture subprocess output** — write `claude --print` stdout/stderr to
   a per-run log file so failures can be investigated post-hoc.
5. **A/B with the legacy path** — re-enable `onshape-cadgen` temporarily
   and run the same 5 prompts × N runs through it. Numbers will be
   apples-to-apples even if the legacy fails on complex prompts.

### Decision

The MCP path is the right architecture given the failure mode of the
legacy and the iteration model it enables — but its actual speed advantage
in interactive use (where session persistence matters) was **not** validated
by this benchmark. A realistic interactive benchmark requires modeling a
user with a long-lived Claude session, not subprocess-per-prompt. That
work is logged for follow-up.
