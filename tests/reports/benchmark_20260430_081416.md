## Analysis

### What worked

**P1 (simple cube + hole)** and **P2 (L-bracket + 4 holes)** both
completed successfully and produced correct geometry. Claude self-
verified its work with measurement tools and reported a check-table.
Examples from the logs:

> P1: "Cube size 30 × 30 × 30 mm, bbox ±15 mm all axes ✓ … Volume 24 643 mm³ = 27 000 − π·5²·30 ✓"
> P2: "Bounding box 60.00 × 40.00 × 34.00 mm ✓ … Volume delta −145.3 mm³ = 4 × π × 1.7² × 4 (analytically correct) ✓"

The session-resume mechanic (`--resume <uuid>` for P4 after `--session-id
<uuid>` for P3) **technically works** — P4 picked up where P3 left off
and added 6 more features. The bug from the v2 first attempt
("`--session-id` is already in use") was fixed by switching follow-ups
to `--resume`.

### What didn't

**P3 (snap-fit enclosure)** added 5 features in 600 s and was
stopped by the timeout. **P5 (FeatureScript gear)** added 0 features
in 600 s. Both subprocess logs are empty past the header — `claude
--print` only writes its final response on completion, so timeout =
no captured output, no insight into where the model got stuck.

### The bottleneck is LLM per-feature latency, not the MCP plumbing

Per-feature rates from the successful runs:

| Prompt | Features | Wall | s/feature |
|--------|---------:|-----:|----------:|
| P1 simple cube | 4 | 66.9 s | **16.7 s** |
| P2 L-bracket | 4 | 446.5 s | **111.6 s** |
| P3 enclosure (partial) | 5 | 600.0 s | **120.0 s** |
| P4 iter (partial) | 6 | 600.0 s | **100.0 s** |

P1's 16.7 s/feature is the floor — minimal context, simple geometry.
P2 jumps almost 7× to 111 s/feature once "verify the holes are
positioned correctly" reasoning kicks in. P3/P4 stay around 100–120 s/
feature, consistent with the inspection + reasoning overhead. P5 produced
zero features — Sonnet appears to struggle with FeatureScript synthesis
at this size (most likely stuck in a draft-and-retry loop trying to
write a parametric gear FS feature).

This is a **model latency problem**, not an MCP/Onshape plumbing problem:

- Onshape REST tool calls themselves are sub-second (uvx mcp-proxy →
  Jarvis → Onshape API → response).
- The Jarvis MCP server stays warm and does not crash.
- Microversion advances correctly even on partial completion (P3 added
  5 features, P4 added 6 — geometry is being created reliably,
  just not finishing the requested spec).

### Implications for the architecture

The MCP path is the right architecture, and it works. But:

1. **Sonnet is too slow for non-trivial Onshape design.** Per-feature
   times scale poorly with complexity. Use **Opus** (Claude Code default)
   for real interactive design work. This benchmark used Sonnet because
   `--model sonnet` is the cheap default for `claude --print` — a
   real interactive user on Max/Pro gets Opus.

2. **`claude --print` is the wrong invocation pattern for this work.**
   It pays a full subprocess cold-start, has no streaming output, and
   times out invisibly. The right pattern is one persistent `claude`
   session per design conversation, where the user can see partial
   progress and intervene.

3. **The v1 60 s P1 measurement is real.** The architectural overhead
   per turn (uvx cold ~2 s + LLM thinking + Onshape REST) is
   unavoidable at this latency floor for fresh subprocesses.
   Persistent-session use will be much faster on follow-ups (cache hit
   on context) — not measured here.

4. **The legacy iframe was faster on simple parts** (25–40 s P1 vs
   66 s here) but the per-feature scaling cliff hits both paths. The
   legacy path's cliff is the Translation+Derived feature import
   (which often returns ERROR); the new path's cliff is LLM thinking
   time on multi-feature reasoning. The new path **always converges to
   correct geometry given enough time**; the legacy path **fails
   atomically** on import errors.

### What this benchmark cannot measure

The benchmark is structurally biased against the architecture's
real value:

- **Subprocess-per-prompt** annihilates the iterative session benefit.
- **Sonnet** (default `claude --print`) is used instead of Opus.
- **No partial-progress UI** — a real user sees feature creation live
  and can cancel/redirect at 30 s instead of waiting 600 s.
- **No model warm-cache** — Anthropic prompt caching kicks in on
  follow-ups within 5 min; here every invocation pays full cost.

The realistic interactive case (one Claude Code session, Opus, user
sees progress, prompt caching warm) is the use case the architecture
was designed for, and it is not what this benchmark exercises.

### Decision (still): MCP path

Confirmed even with these latency numbers:

1. **Reliability** — geometry always lands correctly, partial or full.
   No `Derived feature status=ERROR` failure mode.
2. **Iterative compatibility** — `--resume <uuid>` works; in real use
   the user keeps one session open and pays low cost per follow-up.
3. **User's auth** — model calls go through user's subscription.
4. **Composability** — Variable Studios, mates, FeatureScript, render
   verification all in one toolbox.

For interactive use: keep `claude` open, use Opus, expect ~10–20 s for
simple turns and ~1–3 min for complex multi-feature ones. The
architecture supports that.
